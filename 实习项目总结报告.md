## 三、项目一：AI 训练数据标准化流水线

### 3.1 项目概述

- **仓库地址：** [https://github.com/RXH99/data-pipeline-clean](https://github.com/RXH99/data-pipeline-clean)
- **目标：** 构建一条可复现的 AI 训练数据采集与处理流水线，覆盖从 CSV 数据清洗到 AI 编程助手交互日志标准化的完整链路，每条输出都经过一致性校验。

### 3.2 系统架构

```
┌──────────────────────────────────────────────────┐
│             pipeline/converter.py                 │
│  CSV → 缺失值填充 → 列筛选 → UUID注入 → JSONL   │
└──────────────────────┬───────────────────────────┘
                       ↓
┌──────────────────────────────────────────────────┐
│           agent_recorder/recorder.py              │
│                                                    │
│  snapshot_initial_state()                         │
│    ├─ 记录 Git commit + 文件内容 SHA256            │
│    └─ 捕获环境快照                                 │
│                                                    │
│  record() / record_run()                          │
│    ├─ 记录每一步交互（思考/读文件/编辑/运行/验证） │
│    └─ 每步带时间戳                                 │
│                                                    │
│  finalize()                                       │
│    ├─ 生成 Unified Diff                           │
│    └─ 输出完整 JSONL（含交互轨迹 + 文件快照）      │
└──────────────────────┬───────────────────────────┘
                       ↓
┌──────────────────────────────────────────────────┐
│         ConsistencyValidator                     │
│  SHA256 哈希校验 + difflib 差异验证               │
│  不依赖 git apply（避免 Windows CRLF 问题）       │
└──────────────────────────────────────────────────┘
```

### 3.3 功能特性

| 功能 | 说明 |
|------|------|
| CSV → JSONL 转换 | 支持缺失值填充（均值/中位数/固定值）、列筛选、UUID 注入 |
| Agent 交互记录 | 模拟 AI 编程助手工作流程，自动记录从需求分析到验证的完整步骤 |
| 一致性校验 | SHA256 哈希 + difflib 双重验证 initial + patch → final |
| Docker 可复现 | Dockerfile + docker-compose.yml 一键构建运行环境 |
| 质量报告 | 自动输出 JSONL 格式的交互记录 + quality_report.json |

### 3.4 项目结构

```
data-pipeline-clean/
├── pipeline/converter.py        # CSV→JSONL 转换器
├── agent_recorder/
│   ├── recorder.py              # 交互记录器 + 一致性校验器
│   └── tasks.py                 # 预置任务模板
├── scripts/run_agent_demo.py    # 一键运行全流程
├── tasks/                       # 运行时生成的任务文件
├── output/                      # 运行产出（orders.jsonl + agent_records.jsonl + quality_report.json）
├── Dockerfile
└── docker-compose.yml
```

### 3.5 典型使用流程

```bash
# 全流程演示（转换 → 记录 → 校验 → 报告）
python scripts/run_agent_demo.py

# 输出文件
output/
├── orders.jsonl              # CSV 转换结果（100 行）
├── agent_records.jsonl       # Agent 交互轨迹（含 Unified Diff）
└── quality_report.json       # 一致性校验报告
```

### 3.6 与旧版项目的区别

之前的 `csv-to-jsonl-tool` 只有基础 CSV 转换功能。新版 `data-pipeline-clean` 新增了：

- 分层架构（pipeline / agent_recorder / scripts 分离）
- Agent 交互轨迹自动化记录（不再是手动写日志）
- 一致性校验器（纯 Python 实现，不依赖 git apply）
- 质量报告自动输出

### 3.7 遇到的问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 一致性校验 corrupt patch | Windows 系统 CRLF 换行符与 git apply 不兼容 | 改用纯 Python 校验（SHA256 + difflib） |
| `_generate_unified_diff` 缩进错位 | 用 Notepad 编辑时缩进丢失 | 统一用 VS Code 查看缩进 |
| 验证器提示缺少 diff | git diff 默认不追踪新文件 | 改用 git diff --cached（先 add 再 diff） |

## 四、项目二：RAG 检索效果对比平台

### 4.1 项目概述

- **仓库地址：** [https://github.com/RXH99/rag-bench](https://github.com/RXH99/rag-bench)
- **目标：** 搭建一个可配置的 RAG 问答系统，并设计对比实验评测不同 chunk_size 和检索策略的效果差异。

### 4.2 系统架构

```
                    ┌──────────────────┐
                    │  本地技术文档 .txt  │
                    │  4 篇，共 5000 字  │
                    └────────┬─────────┘
                             ↓
                    ┌──────────────────┐
                    │   文档切片 (Chunk) │
                    │   支持 3 种大小    │
                    └────────┬─────────┘
                             ↓
                    ┌──────────────────┐
                    │  向量嵌入 (Chroma) │
                    │ nomic-embed-text  │
                    └────────┬─────────┘
                             ↓
         ┌───────────────────────────────────────┐
         │          用户问题                      │
         └────────┬──────────────────────────────┘
                  ↓
    ┌───────────────────────────┐
    │  对比 2 种检索策略        │
    │  Top-K（相似度）/ MMR     │
    └───────────┬───────────────┘
                ↓
         ┌──────────────────┐
         │  qwen:4b 生成回答  │
         └────────┬─────────┘
                  ↓
    ┌───────────────────────────────┐
    │  CLI / Streamlit Web 双界面    │
    └───────────────────────────────┘
```

### 4.3 功能特性

- **文档加载：** 加载 `data/` 目录下所有 `.txt` 文件，纯 Python 实现，无弃用警告
- **向量索引构建：** 使用 `nomic-embed-text` 嵌入模型，支持按 chunk_size 缓存不同版本
- **RAG 问答：** 检索 → 增强 → 生成全流程，支持运行时切换 Top-K / MMR
- **Benchmark 对比：** 自动测试 6 种配置组合，输出命中率对比表
- **双界面支持：** 命令行（`python -m scripts.query`）+ Streamlit Web（`streamlit run scripts/app.py`）

### 4.4 项目结构

```
rag-bench/
├── rag/                         # 核心 RAG 库
│   ├── pipeline.py              # RAGPipeline 类（加载 → 分割 → 向量化 → 检索 → 问答）
│   ├── config.py                # 集中配置管理
│   └── loader.py                # 文档加载器（纯 Python）
├── eval/
│   ├── benchmark.py             # 对比实验（6 种配置）
│   └── questions.json           # 15 个 QA 测试对
├── scripts/
│   ├── query.py                 # 交互式问答
│   ├── build_db.py              # 构建向量库
│   └── app.py                   # Streamlit Web 界面
├── data/                        # 知识库文档
└── results/                     # Benchmark 报告
```

### 4.5 Benchmark 结果

| 配置 | Hit Rate | Top-1 Precision |
|------|----------|-----------------|
| cs=300, Top-K | 26.67% | 15.00% |
| cs=300, MMR | 20.00% | 15.00% |
| cs=500, Top-K | 40.00% | 32.33% |
| cs=500, MMR | 26.67% | 32.33% |
| cs=1000, Top-K | **53.33%** | **31.11%** |
| cs=1000, MMR | 40.00% | 31.11% |

**结论：**
- chunk_size=1000 + Top-K 命中率最高（53.33%）
- chunk_size 越大效果越好（技术文档的语义单元较完整）
- 当前数据量下 Top-K 优于 MMR（多样性收益不明显）

### 4.6 与旧版项目的区别

之前的 `local-rag-demo` 只有基础问答功能。新版 `rag-bench` 新增了：

- 分层模块架构（rag/eval/scripts 分离）
- 集中配置管理（所有参数通过 dataclass 管理）
- Benchmark 对比实验（自动运行 6 种配置）
- 纯 Python 加载器（避免 langchain-community 弃用警告）
- 完整的 README 和 Docker 支持

### 4.7 遇到的问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 第一次 Benchmark 命中率 6.67% | 向量库缓存了旧文档，新文档数据未更新 | 删除 chroma_db_eval_* 缓存重建 |
| 所有配置命中率相同 | 旧文档数据太少（4个文件不到1KB） | 扩展文档到 4 篇共 5000 字的技术笔记 |
| `langchain-community` 弃用警告 | LangChain 社区版已 sunset | 改用纯 Python 写文档加载器 |

## 五、项目三：OpenClaw AI Agent 部署

### 5.1 项目概述

- **目标：** 在 WSL (Ubuntu-22.04) 环境中安装并配置 OpenClaw AI Agent 框架，实现 Web UI 和 TUI 的双界面管理。

### 5.2 安装与配置流程

1. 在 WSL (Ubuntu-22.04) 中执行官方安装脚本
2. 自动安装 Node.js 24 + npm，全局安装 OpenClaw
3. 配置 DeepSeek API 作为后端模型
4. 启动网关，访问 Web UI 和 TUI 双界面

### 5.3 技术栈

| 组件 | 版本/说明 |
|------|-----------|
| OpenClaw | 2026.5.28 |
| Node.js | v24.15.0 |
| 操作系统 | WSL 2 Ubuntu-22.04 |
| 后端模型 | DeepSeek API（在线）/ qwen:4b（本地尝试） |

### 5.4 遇到的问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| GitHub 克隆失败 | 网络环境限制 SSH 22 端口 | 配置 SSH 改用 443 端口转发 |
| npm 全局安装权限不足 | 默认安装到系统目录 | 安装脚本自动切换为用户级目录 `~/.npm-global` |
| dashboard 未输出 token | 命令执行问题 | 从配置文件手动提取 token |
| 切换到 Ollama 导致配置损坏 | 配置格式不兼容 | 恢复备份配置文件 |
| Web UI 响应慢 | 跨境网络延迟 | 接受为正常现象 |

| OllamaEmbeddings 导入变更 | 项目二 | 更新为 langchain_ollama 新路径 |

| 问题 | 涉及项目 | 解决方案 |

| 向量库缓存未更新 | 项目二 | 删除旧 chroma_db 目录重建 |
